# Three-Minute Video Script

Target length: 2:45-2:55.

Tone: human, urgent, concrete. The video should feel like a real product saving time in a bad situation, not a feature tour.

## Storyboard

| Time | Visual | Voiceover |
| --- | --- | --- |
| 0:00-0:15 | Dark forest / damaged street / no-signal phone icon. | "In a disaster, the cloud is often the first thing you lose. But your phone may still have battery." |
| 0:15-0:30 | Beacon home screen, large simple actions. | "Beacon turns that phone into an offline survival assistant powered by Gemma 4 running on-device." |
| 0:30-0:50 | Settings & Models, Gemma 4 E2B downloaded and ready. Toggle airplane mode. | "After the model is downloaded, Beacon does not need a server, API key, or network." |
| 0:50-1:25 | User asks: "I am lost in a forest..." Answer streams. | "The app retrieves local survival guidance, then Gemma 4 turns it into immediate steps: stop moving randomly, protect warmth, mark location, conserve water, and prepare signals." |
| 1:25-1:50 | Ask a follow-up: "Should I keep walking?" | "Beacon keeps the current emergency context, so follow-up questions stay grounded in the same situation." |
| 1:50-2:10 | Visual help: take/select photo, image appears in chat, response references it. | "If the user has an injury, unknown plant, or hazard, they can send a photo. The image stays on the device." |
| 2:10-2:30 | Switch language to Chinese or Arabic RTL. | "Beacon is built for ordinary people under stress, in 20 UI languages, including right-to-left Arabic." |
| 2:30-2:45 | Runtime diagnostics and architecture diagram. | "Under the hood: React, Capacitor, Kotlin, Google AI Edge LiteRT-LM, Gemma 4, and an offline emergency knowledge base." |
| 2:45-2:55 | Logo + final line. | "Beacon: survival intelligence for when the network is gone." |

## Required Proof Shots

- GitHub Release page with APK.
- App installed on a real Android phone.
- Airplane mode enabled before asking a question.
- Model status showing Gemma 4 E2B/E4B ready.
- Answer streaming, not appearing as a fixed template.
- Runtime diagnostics showing local runtime/model.
- Photo visible in chat before visual answer.

## Recording Rules

- Use a real phone recording, not only a desktop mock.
- Keep one continuous sequence for the offline proof.
- Blur personal photos, device identifiers, and location.
- Do not show private model tokens, keystore details, or local credentials.
- Avoid medical claims such as "diagnoses" or "guarantees survival."
